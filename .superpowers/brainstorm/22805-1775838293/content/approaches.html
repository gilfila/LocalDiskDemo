Three Architectural Approaches

How should we structure the pages and orchestration for the parking registration app?

A

Single Hub Page

One smart home page that shows/hides sections based on registration status. Manager view is a second page. Fewer files, simpler navigation.


Pages: Home (worker smart page) → Manage Commuters (manager page)
Registration handled inline via form sections that appear/disappear
✓ Minimal file count   ✗ Complex conditional logic in one PMD
B

Multi-Page Wizard Flow Recommended

Dedicated pages for each step. Worker flow is a 3-step wizard. Manager has a separate commuter table page. Clean separation, standard Workday Extend patterns.


Pages: Home/Status → Register Car (wizard: Details → Review → Confirm) → Manage Commuters
Each page has one clear responsibility
✓ Clean UX   ✓ Easier to extend   ✓ Proper Workday patterns
C

Tabbed Navigation Portal

Full portal with tabbed nav. Workers have "My Registration" and "My Benefits" tabs. Managers see "Team" and "OTP History" tabs. Most polished, most complex.


Pages: Worker Portal (2 tabs) + Manager Portal (2 tabs)
Richer UX with benefit history, OTP log visible to workers
✓ Most complete   ✗ Higher complexity   ✗ More orchestrations needed

All approaches include: Auto distance check via Workday home address API · Manager approval business process · Anniversary-based scheduled OTP ($100) · Ad-hoc OTP trigger for managers
